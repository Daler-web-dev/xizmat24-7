# XIZMAT24 — Admin Seeding Mini App · Build Spec

> Документ для Claude Code. Цель: построить **Telegram Mini App на Next.js**, через который админ вручную заносит специалистов на платформу XIZMAT24, плюс поднять под это базу в Supabase. Деплой — Vercel. Backend платформы и мобильное приложение делаются ОТДЕЛЬНО и ПОЗЖЕ — этот инструмент их не касается. Но БД проектируется сразу боевой, чтобы потом не мигрировать.

---

## 0. ВАЖНО — границы scope (читать первым)

Это **тонкий admin-инструмент ввода данных**, а не платформа.

- Server actions / route handlers ходят **прямо в Supabase**. Это НЕ «бекенд платформы».
- НЕ создавать здесь NestJS, отдельный API-слой, каталог для пользователей, поиск для конечных юзеров, аутентификацию специалистов и т.п.
- Платформенный backend (NestJS + Prisma) и мобильное приложение — отдельный проект позже.
- Общей с будущим остаётся **только схема БД** — она сразу боевая. Код Mini App — это admin-тула, и точка.

Если ловишь себя на мысли «раз я уже на Next.js, допишу заодно…» — НЕ дописывай. Любое расширение за пределы ввода данных админом = вне scope.

## 1. Контекст и принципы

**Что это.** XIZMAT24 — платформа, где люди находят специалистов (сантехников, электриков, репетиторов, мастеров красоты и т.д.) и их контакты. Приложение ещё не готово. Нужны сиды — реальные специалисты в базе. Админ ходит, знакомится с людьми и заносит их через эту Mini App с телефона.

**Роль инструмента.** НЕ публичный продукт. Полевой инструмент ввода для админа и пары доверенных операторов. Доступ — по whitelist Telegram ID, проверяется на сервере после валидации initData.

**Ключевые принципы (строго):**

1. **Телефон = будущий логин.** Позже специалист войдёт в приложение по своему номеру и сам дозаполнит профиль. Номер обязан быть строго нормализован (`+998XXXXXXXXX`) и уникален. Единственное поле без права на ошибку.
2. **Сейчас заносим только минимум.** Остальные поля в схеме есть, но nullable — их заполнит сам специалист через приложение. Mini App их не спрашивает.
3. **Гео и категории — только из справочников** (дропдауны/мультиселект), никогда свободным текстом.
4. **Рейтинг руками НЕ заносится** — вычисляется из отзывов. На сидах рейтинга нет (null).
5. **БД проектируется на вырост сразу.** Все будущие поля заложены nullable, миграций потом быть не должно.

## 2. Технологический стек

- **Framework:** Next.js (App Router, TypeScript)
- **Telegram Mini App SDK:** `@telegram-apps/sdk-react` (получение и проброс `initData` на клиенте)
- **Хостинг:** Vercel (бесплатный tier, serverless — Mini App для этого подходит идеально, в отличие от polling-бота)
- **DB:** Supabase (PostgreSQL)
- **DB-клиент:** `@supabase/supabase-js`, вызывается **только в серверном коде** (server actions / route handlers) под **service_role**-ключом
- **Телефоны:** `libphonenumber-js` (валидация по региону `UZ`)
- **UI:** Tailwind CSS; компоненты под Telegram-тему (использовать CSS-переменные темы Telegram `--tg-theme-*`)
- **Валидация форм:** `zod` (одна схема валидации на клиенте и в server action)

**Точка входа для пользователя:** бот в @BotFather, у которого настроена кнопка меню (Menu Button) или inline-кнопка, открывающая URL этой Mini App. Сам бот — пустая оболочка, никакого процесса хостить не надо, вся логика в Next.js на Vercel.

## 3. Схема базы данных (Supabase / Postgres)

> Идентична независимо от стека. Создать как миграцию `/supabase/migrations/0001_init.sql`. Иерархия категорий **двухуровневая**: 10 главных → подкатегории. Специалист привязан к **подкатегориям** (листьям); главная выводится через родителя.

### 3.1 Справочники (заполняются сидами один раз)

```sql
create table regions (
  id          serial primary key,
  name_ru     text not null,
  name_uz     text not null,
  sort_order  int  not null default 0
);

create table tumans (
  id          serial primary key,
  region_id   int  not null references regions(id) on delete restrict,
  name_ru     text not null,
  name_uz     text not null,
  sort_order  int  not null default 0
);
create index idx_tumans_region on tumans(region_id);

create table categories (
  id          serial primary key,
  name_ru     text not null,
  name_uz     text,
  sort_order  int  not null default 0
);

create table subcategories (
  id          serial primary key,
  category_id int  not null references categories(id) on delete restrict,
  name_ru     text not null,
  name_uz     text,
  sort_order  int  not null default 0
);
create index idx_subcategories_category on subcategories(category_id);
```

### 3.2 Боевые таблицы

```sql
create table workers (
  id            uuid primary key default gen_random_uuid(),

  -- Заносит Mini App (минимум на этапе сидов):
  name          text not null,
  phone         text not null unique,          -- строго +998XXXXXXXXX, ЛОГИН на будущее
  region_id     int  references regions(id) on delete set null,
  tuman_id      int  references tumans(id)  on delete set null,
  rate          numeric,
  rate_type     text check (rate_type in ('day','task','hour') or rate_type is null),

  -- Служебное (ставит сервер автоматически):
  added_by      bigint not null,                -- Telegram ID админа (из initData)
  status        text not null default 'seeded', -- seeded | active
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now(),

  -- На вырост: заполнит специалист через приложение (nullable, Mini App НЕ трогает):
  avatar_url    text,
  bio           text,
  experience    text,
  language      text,
  rating        numeric,                        -- КЭШ из reviews (AVG); на сидах null
  priority      int default 0                   -- ручной флаг "показывать выше"
);
create unique index idx_workers_phone on workers(phone);

create table worker_subcategories (
  worker_id      uuid not null references workers(id) on delete cascade,
  subcategory_id int  not null references subcategories(id) on delete restrict,
  primary key (worker_id, subcategory_id)
);

create table worker_portfolio (
  id          uuid primary key default gen_random_uuid(),
  worker_id   uuid not null references workers(id) on delete cascade,
  url         text not null,
  sort_order  int  not null default 0,
  created_at  timestamptz not null default now()
);

create table reviews (
  id          uuid primary key default gen_random_uuid(),
  worker_id   uuid not null references workers(id) on delete cascade,
  rating      int  not null check (rating between 1 and 5),
  comment     text,
  author      text,
  created_at  timestamptz not null default now()
);
create index idx_reviews_worker on reviews(worker_id);
```

Включить RLS на боевых таблицах. Mini App ходит под **service_role** из серверного кода → RLS обходит. Политики на старте закрытые — задел под будущее приложение (anon/authenticated с правильными политиками). Справочники (`regions`, `tumans`, `categories`, `subcategories`) можно открыть на `select` для anon.

## 5. Авторизация (Telegram initData)

Это главное отличие Mini App от обычного бота — здесь нет «просто проверю Telegram ID». Поток:

1. **Клиент:** через `@telegram-apps/sdk-react` получить `initDataRaw` (подписанная строка от Telegram). Прикладывать её к каждому запросу к серверу (заголовок `Authorization: tma <initDataRaw>` или в теле server action).
2. **Сервер:** на каждом защищённом вызове **валидировать `initData`**:
   - Пересобрать data-check-string, посчитать HMAC-SHA256 с ключом, производным от `BOT_TOKEN` (стандартный алгоритм Telegram Mini Apps), сравнить с `hash`.
   - Проверить `auth_date` на свежесть (отвергать старые).
   - Если подпись невалидна → 401, ничего не делать.
3. **Whitelist:** из валидного initData достать `user.id`, проверить, что он в `ADMIN_IDS`. Если нет → 403.
4. Только после (2) и (3) выполнять любую работу с БД.

> Реализовать валидацию initData как переиспользуемую серверную функцию (`lib/auth/validateInitData.ts`), оборачивать ею каждый server action / route handler. Никогда не доверять `user.id`, пришедшему с клиента без проверки подписи.

**Безопасность ключей:** `SUPABASE_SERVICE_ROLE_KEY` и `BOT_TOKEN` — только серверные env (без префикса `NEXT_PUBLIC_`). Supabase-клиент инициализировать исключительно в серверных модулях. Если service_role попадёт в клиентский бандл — это дыра в боевую БД.

## 6. Экраны и функциональность

Mini App = несколько экранов внутри Telegram. Минимализм, мобайл-фёрст, под тему Telegram.

### 6.1 Главный экран
- Кнопка **«+ Добавить специалиста»**.
- Список последних добавленных (10): имя, телефон, категории, регион/туман. Тап → карточка.
- Поле поиска (по имени/телефону).
- Если юзер не в whitelist → экран отказа.

### 6.2 Форма добавления специалиста

Одна форма (можно в несколько секций/шагов), валидация через `zod`:

| Поле | Контрол | Правила |
|---|---|---|
| Имя | text input | required, trim |
| Телефон | tel input + кнопка «Взять мой контакт» (Telegram `requestContact`) | required, нормализация `+998XXXXXXXXX`, проверка дубля |
| Специальности | **searchable мультиселект** по подкатегориям (с группировкой по главной категории) | минимум 1; можно несколько — сантехник + кондиционерщик |
| Регион | searchable select | required |
| Туман | searchable select, **зависимый от региона** (каскад) | required, фильтруется по выбранному региону |
| Ставка | number + сегмент тип (день/задача/час) | опционально |

- **Мультиселект специальностей** — ключевое преимущество формы над ботом: ввёл «санте» → выбрал, ввёл «конди» → выбрал. Показывать выбранные чипами с удалением.
- **Каскад регион→туман:** при смене региона — сбрасывать туман, грузить только его туманы.
- Перед сабмитом — экран подтверждения с собранной карточкой.
- Сабмит → server action.

### 6.3 Server action: создание специалиста (транзакционно)
1. Валидировать initData + whitelist.
2. Валидировать payload через `zod`.
3. Нормализовать телефон (`libphonenumber-js`, region `UZ`), проверить `.isValid()`.
4. Проверить уникальность телефона. Если есть → вернуть существующую запись, НЕ создавать дубль.
5. INSERT `workers` (`status='seeded'`, `added_by` = id из initData) + INSERT `worker_subcategories` (все выбранные) — атомарно (через Postgres-функцию/RPC или две операции с откатом при ошибке).
6. Вернуть результат → экран «✅ Сохранён».

### 6.4 Поиск / список / редактирование / удаление
- **Поиск:** server action по `name ILIKE` или нормализованному телефону.
- **Карточка специалиста:** все поля + кнопки «Редактировать» / «Удалить».
- **Редактирование:** та же форма, предзаполненная; апдейт через server action (с теми же проверками).
- **Удаление:** подтверждение → server action (каскад снесёт `worker_subcategories`).

## 7. Нормализация и валидация телефона

- Принимать любой ввод: `+998901234567`, `998 90 123 45 67`, `90 123 45 67`.
- `parsePhoneNumberFromString(input, 'UZ')` → проверить `.isValid()` → хранить `.number` (E.164 = `+998XXXXXXXXX`).
- Отклонять всё, что не проходит `.isValid()` для UZ.
- Проверка уникальности и на клиенте (UX), и на сервере (правда).

---

## 8. Edge-кейсы

- **Дубль телефона** — не создавать второй, показать существующего.
- **Кривой номер** — ошибка валидации, не сабмитить.
- **Невалидный/просроченный initData** — 401, экран «Откройте через Telegram».
- **Не-whitelisted** — 403, экран отказа.
- **Сбой записи** — атомарность worker + subcategories, при ошибке откат и понятное сообщение.
- **Повторный запуск сидов** — идемпотентность.
- **Тема Telegram (тёмная/светлая)** — использовать `--tg-theme-*`, не хардкодить цвета.

## 11. Деплой (Vercel)

1. Залить репозиторий на Vercel, прописать env (см. выше) в настройках проекта.
2. В @BotFather: создать бота, через `/setmenubutton` (или BotFather → Bot Settings → Menu Button) указать URL задеплоенной Mini App.
3. Mini App открывается из чата с ботом по кнопке меню.

### Резюме для исполнителя

Тонкая admin Mini App на Next.js поверх боевой Supabase-схемы, деплой на Vercel. Критичны: (1) валидация initData + whitelist на сервере перед любым доступом к БД; (2) service_role-ключ только в серверном коде; (3) телефон нормализован и уникален — будущий логин; (4) категории и гео только из справочников; (5) схема на вырост, миграций потом быть не должно. Всё остальное — минимализм. Не превращать admin-тулу в платформенный бекенд.
